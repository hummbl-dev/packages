# v0.1 Boundary

> Status: **Non-canon** — describes the scope and adoption boundary of the
> Distribution Spine v0.1 proposal.

## Scope

The Distribution Spine v0.1 establishes:

1. **Canonical package identity** — a registry of packages owned by HUMMBL
   (`index/package-identity-registry.json`), with this repository as the
   canonical source.
2. **Canonical release metadata format** — the fields that constitute a
   canonical release artifact receipt
   (`schemas/release-artifact-receipt-v0.1.json`): version, artifact URL,
   sha256, SBOM pointer, provenance pointer, release status, and generated
   surfaces.
3. **Downstream surface relationship** — the rule that install surfaces
   (homebrew-tap, scoop-bucket, winget-manifests, nix) are generated from and
   validated against the canonical metadata, and never independently define
   artifact identity.
4. **Governed release pipeline** — the principle that one release triggers
   deterministic generation of all install surfaces, with a receipt chain
   linking the release to each surface.

## Non-Goals

The v0.1 boundary explicitly does **not** include:

- **Build pipeline internals** — how individual packages are built is out of
  scope; the spine begins at the released artifact.
- **Signing key management** — key rotation, storage, and trust policy are
  deferred to a later version.
- **End-user installation workflows** — the spine governs artifact identity and
  surface generation, not the user-facing install experience.
- **Surface-specific formatting** — the exact format of a Homebrew formula,
  Scoop manifest, Winget manifest, or Nix derivation belongs in the respective
  downstream repositories, validated against the spine.
- **Multi-architecture matrix policy** — handling of per-arch artifacts is
  deferred.
- **Channel / rolling release semantics** — only `released`, `prerelease`, and
  `yanked` statuses are modeled in v0.1.

## Adoption Boundary

The v0.1 artifacts are **non-canon**. They describe a proposed architecture.
Adoption occurs when HUMMBL governance explicitly promotes a versioned record
to canon, at which point:

- This repository becomes the binding source of truth for the modeled packages.
- Downstream surfaces become required to consume and validate against the spine.
- The governed release pipeline becomes the required path for publishing
  releases.

Until adoption, the v0.1 artifacts may be referenced, iterated, and staged, but
they impose no obligations on downstream repositories or existing release
processes.

## Versioning

The v0.1 artifacts are versioned as a coherent set. Substantive changes to any
artifact in the set produce a v0.2 (or later) set rather than in-place edits,
so that the proposal history remains traceable.
